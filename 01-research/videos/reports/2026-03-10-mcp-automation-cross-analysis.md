## MCP vs n8n 비교 분석

| 기준 | MCP 접근법 | n8n 접근법 |
|------|-----------|-----------|
| **패러다임** | AI가 외부 서비스를 직접 호출하는 "AI-first" 방식 | 사전 정의된 조건 분기로 워크플로우를 오케스트레이션하는 "flow-first" 방식 |
| **강점** | 자연어 지시만으로 다단계 작업 실행, 맥락 이해 및 창의적 판단 가능 | 24시간 서버 운영, 예측 가능한 반복 실행, 에러 시각화, 구조 파악 용이 |
| **약점** | 세션 의존성, 비용 발생, 결과 비일관성, 서버 상태 유지 어려움 | 복잡한 맥락 이해 불가, "경우의 수"가 많은 창의적 작업에 취약 |
| **적합 작업** | 콘텐츠 생성, 분석, 창의적 판단, 비정형 데이터 처리 | 데이터 전달, 서비스 간 연동, 스케줄 트리거, 반복 조건 분기 |
| **설정 난이도** | JSON 설정 + API 키 발급 (서비스별 복잡도 편차 큼) | 노드 기반 시각적 편집기 (복잡한 로직은 학습 필요) |
| **토큰/비용** | MCP 서버 상시 로드 시 도구 정의만으로 토큰 소비, 실행마다 API 비용 | 자체 호스팅 시 실행 비용 분리 가능, AI 호출 시에만 API 비용 발생 |
| **안정성** | 프로토콜 초기 단계, 서버별 안정성 편차 큼 | 검증된 프로덕션 레이어, 에러 추적과 수정 용이 |
| **우리 시스템 현황** | Notion, Filesystem, Sequential Thinking, Playwright CLI 등 다수 설치·운영 중 | 미도입. cron + Subagent 조합이 오케스트레이션 역할 대체 중 |

**핵심 구분선 (교차 분석에서 발견):** MCP는 "AI가 도구를 사용하는 방식"이고, n8n은 "AI를 도구로 사용하는 방식"이다. 두 접근법은 방향이 반대이며, 이것이 경쟁이 아닌 보완 관계의 근거다.

## 클러스터 종합 인사이트

개별 영상에서는 보이지 않지만 교차 분석에서 발견된 인사이트:

**1. MCP와 자동화 클러스터가 수렴하는 "레이어드 자동화 아키텍처"가 존재한다**

MCP 클러스터 영상들은 "AI가 도구를 사용하는 방법"을, 자동화 클러스터 영상들은 "AI 작업을 언제 어떻게 실행할 것인가"를 다룬다. 두 클러스터를 합치면 실용적인 3계층 아키텍처가 도출된다.

```
계층 1 (오케스트레이션): cron / n8n / /loop / GitHub Actions → "언제, 어떤 조건에 실행할지"
계층 2 (실행): Claude Code + MCP 서버 → "무엇을 어떻게 처리할지"
계층 3 (도구): Notion MCP, Playwright CLI, NanoBanana 등 → "어떤 외부 자원을 활용할지"
```

우리 시스템은 이미 이 3계층을 운영 중이나, 계층 간 역할 분리가 문서화되지 않았다. 이것이 가장 중요한 갭이다.

**2. "AI를 도구로 볼 것인가, AI가 도구를 쓸 것인가"의 패러다임 전환이 진행 중이다**

n8n 클러스터 영상들은 n8n이 Claude API를 노드로 호출하는 구조(AI를 도구로)를 보여주고, MCP 클러스터 영상들은 Claude가 n8n MCP를 통해 워크플로우를 직접 생성하는 구조(AI가 도구를 사용)를 보여준다. 두 방향이 공존하며 각각의 최적 사용 사례가 다르다. 단방향적으로 하나를 선택하는 것은 손해다.

**3. 토큰 효율성과 자동화 신뢰성은 트레이드오프 관계이다**

MCP 서버를 많이 설치할수록 도구 정의 토큰이 증가하고, `/loop`를 짧은 간격으로 실행할수록 컨텍스트가 누적된다. Playwright CLI가 MCP 대비 9만 토큰을 절약하는 것은 이 트레이드오프를 의식적으로 선택한 결과다. **"자동화 추가 = 컨텍스트 부담 증가"라는 공식이 모든 영상에 암묵적으로 흐르고 있으며, 이것이 Skill 패키징과 표준화가 중요한 이유다.**

**4. 안정성 계층구조가 자동화 도구 선택의 실질적 기준이다**

영상들을 종합하면 자동화 도구의 안정성은 아래 순서를 따른다.

```
가장 안정적: 외부 cron + shell script
              ↓
          n8n (검증된 운영 레이어)
              ↓
          GitHub Actions (인프라 의존)
              ↓
          Claude Desktop /schedule
              ↓
          Claude Code /loop (세션 의존)
              ↓
가장 불안정: MCP 서버 (프로토콜 초기 단계)
```

프로덕션 크리티컬한 자동화일수록 스택의 위쪽을 선택해야 한다.

**5. 컨텍스트 문서(CLAUDE.md, Skills, Agents)가 병렬화의 품질 안전망이다**

QKoSVY2T4ec에서 5개 병렬 에이전트가 일관성을 유지한 것은 FIELD/UI 가이드 컨텍스트 문서 덕분이었다. 9aS--Zs5ILA에서 8단계 가이드가 "성공률을 높인다"고 주장하는 것도 같은 원리다. Trine의 CLAUDE.md, Skills, Agents 체계는 이미 이 역할을 수행하고 있으며, 이것이 다른 팀 대비 병렬화의 품질을 지탱하는 구조적 강점이다.
