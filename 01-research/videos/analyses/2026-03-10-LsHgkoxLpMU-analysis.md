## 1. TL;DR

Claude Code의 `/loop` 기능은 자연어 명령으로 최대 3일간 반복 크론 작업을 세션 내에서 생성·실행하는 단기 폴링 도구이며, 장기 지속성이 필요한 경우에는 별도의 예약 작업(Scheduled Tasks) 기능과 역할을 분리하여 사용해야 한다.

## 3. 핵심 포인트

1. [🕐 00:32](https://youtu.be/LsHgkoxLpMU?t=32) **`/loop`는 Claude Code 내장 스킬**: 자연어("10분마다 배포 상태 확인해줘") 또는 슬래시 명령어로 `cron_create` 도구를 호출하여 크론 작업을 등록한다. 터미널·데스크톱 앱·VS Code 확장 모든 환경에서 사용 가능하다.

2. [🕐 01:38](https://youtu.be/LsHgkoxLpMU?t=98) **자연어가 곧 프롬프트 예약**: 루프는 본질적으로 "프롬프트를 예약하는 것"이다. 지정된 간격마다 동일 세션에 해당 프롬프트가 전송되어 실행된다.

3. [🕐 02:21](https://youtu.be/LsHgkoxLpMU?t=141) **컨텍스트 누적이 최대 단점**: 짧은 간격으로 실행될수록 보고서·응답이 같은 세션에 쌓여 토큰 소모가 급증한다. 루프 빈도와 출력 크기를 설계 시 반드시 고려해야 한다.

4. [🕐 02:35](https://youtu.be/LsHgkoxLpMU?t=155) **커스텀 스킬과 루프 조합 가능**: "20분마다 review-pr 스킬 실행해줘"처럼 기존 커스텀 스킬을 루프와 연결하여 반복 자동화 파이프라인을 구성할 수 있다.

5. [🕐 03:03](https://youtu.be/LsHgkoxLpMU?t=183) **일회성 리마인더 지원**: "오후 3시에 알려줘", "45분 후에 알려줘"처럼 단발 리마인더도 동일 기능으로 처리되며, 실행 후 크론이 자동 삭제된다.

6. [🕐 04:19](https://youtu.be/LsHgkoxLpMU?t=259) **세션 격리**: 루프 작업은 세션별로 독립 관리된다. `cron_list`로 현재 세션 작업만 조회 가능하며, 다른 탭(세션)의 작업은 표시되지 않는다.

7. [🕐 05:31](https://youtu.be/LsHgkoxLpMU?t=331) **3종 크론 도구**: `cron_create`(예약) / `cron_list`(목록) / `cron_delete`(취소). "이거 취소해줘"처럼 자연어로도 각 도구가 자동 호출된다.

8. [🕐 05:53](https://youtu.be/LsHgkoxLpMU?t=353) **3일 자동 만료 — 안전 장치**: 루프 생성 시 기본적으로 3일 타이머가 설정되고 이후 자동 삭제된다. 3일 이상 필요하면 루프를 재생성하거나 예약 작업으로 전환이 권장된다.

9. [🕐 06:21](https://youtu.be/LsHgkoxLpMU?t=381) **환경 변수로 스케줄링 비활성화 가능**: 자연어 대화 중 의도치 않게 크론이 생성되는 상황을 방지하기 위해 환경 변수로 스케줄링 기능 전체를 비활성화할 수 있다.

10. [🕐 07:08](https://youtu.be/LsHgkoxLpMU?t=428) **루프 vs 예약 작업 선택 기준**: "지금 당장/오늘" → 루프, "매일/매주 장기 반복" → 예약 작업(Scheduled Tasks). 루프는 캐치업 없음·세션 종료 시 크론 종료, 예약 작업은 캐치업 있음·영구 저장·현재 데스크톱 앱 전용.

## 4. 비판적 분석

### 주장 1: "/loop는 게임 체인저다"
- **핵심 근거**: 사용자 입력 없이 며칠간 AI가 상태를 자율 확인하는 데모 시연
- **근거 유형**: 경험 (발표자 개인 데모, 통제된 환경)
- **한계**: 실제 API 비용·토큰 소모량 미언급. 자막 오인식으로 일부 도구명 불분명
- **반론/대안**: "게임 체인저"는 과장일 수 있다. 세션 종속성·3일 상한·캐치업 부재로 프로덕션 자동화에는 부적합하며, 외부 cron + 쉘 스크립트 조합이 안정성·지속성에서 우위다. 개발자 세션 내 단기 폴링 도구로서의 가치는 실질적이다.

### 주장 2: "어디서든(터미널·앱·VS Code 확장) 사용 가능하다"
- **핵심 근거**: 세 환경에서의 실행 데모 시연
- **근거 유형**: 경험 (라이브 데모)
- **한계**: "어디서든"은 실행 인터페이스의 다양성을 의미할 뿐, 백그라운드 독립 실행이 아님
- **반론/대안**: 세션이 닫히면 크론도 종료된다는 핵심 제약이 충분히 강조되지 않았다. "어디서든 세션을 열어둔 상태에서 사용 가능"이 더 정확한 표현이다.

### 주장 3: "스킬을 루프와 조합하면 강력한 자동화가 된다"
- **핵심 근거**: "20분마다 review-pull 스킬 실행" 패턴 언급
- **근거 유형**: 경험 (데모 중 간단한 언급, 깊은 실증 없음)
- **한계**: 스킬 반복 실행 시 컨텍스트 누적 문제가 현실적 제약이 됨. 구체적인 비용 분석 없음
- **반론/대안**: 컨텍스트가 소진되면 루프가 의도대로 작동하지 않을 수 있다. 스킬 출력을 최소화하거나 파일로 저장하는 패턴이 병행되어야 실용적이다.

### 주장 4: "컨텍스트 관리와 파일 읽기로 나만의 해결 방법을 만들 수 있다"
- **핵심 근거**: 지속성·컨텍스트 문제에 대한 창의적 접근 가능성 언급
- **근거 유형**: 의견 (구체적 방법 미제시)
- **한계**: 구체적인 해결 방법이 전혀 제시되지 않았다
- **반론/대안**: 루프 중 컨텍스트를 효율적으로 관리하려면 상당한 엔지니어링이 필요하다. 후속 예시 없이는 입문자에게 오해를 줄 수 있는 발언이다.

### 주장 5: "루프 스킬이 호출되는지 여부는 중요하지 않다"
- **핵심 근거**: 크론이 생성되기만 하면 루프 스킬 명시적 호출 여부는 무관하다는 설명
- **근거 유형**: 의견 (내부 원리 부분 설명)
- **한계**: 루프 스킬이 모델에 크론 설정 방법을 알려주는 역할을 한다는 점에서 내부 원리 설명으로는 불완전
- **반론/대안**: 사용자 관점에서는 맞는 설명이나, 모델이 크론 생성을 "이해"하는 것은 루프 스킬의 시스템 프롬프트 덕분이므로 작동 원리를 피상적으로만 설명한다.

## 5. 팩트체크 대상

- **주장**: "3일 만료는 안전을 위한 것입니다" | **검증 필요 이유**: Anthropic 공식 설계 의도 확인 필요. 비용 제어 또는 서버 리소스 관리 목적일 가능성도 있으며, 환경 변수로 연장 가능한지도 미확인 | **검증 방법**: Anthropic 공식 Claude Code 문서, GitHub 소스코드, CLI `--help` 옵션 확인

- **주장**: "예약 작업(Scheduled Tasks)은 현재 데스크톱 앱에서만 사용 가능합니다" | **검증 필요 이유**: 영상 녹화 시점 이후 기능 범위가 확장되었을 가능성이 높음. 영상 스스로 "보실 때쯤 변경되었을 수 있다"고 언급 | **검증 방법**: 최신 Claude Code 릴리즈 노트 및 공식 문서 확인 (claude.ai/code)

## 6. 실행 가능 항목

- [ ] **Claude Code 최신 버전 업데이트 및 `/loop` 기능 활성화 확인** — 기능이 보이지 않으면 확장 프로그램/앱 업데이트 필요 (적용 대상: 개발자 환경 설정)

- [ ] **Trine `pr-code-review-gate.md`의 `/loop 2m` 정책 실제 적용 습관화** — PR 생성 직후 `/loop 2m`으로 CI + 리뷰 폴링 시작. 컨텍스트 소모 모니터링 병행 (적용 대상: Trine PR 워크플로우)

- [ ] **루프+스킬 조합 패턴 실험**: `spec-compliance-checker` 스킬을 20분마다 루프 실행하는 패턴 테스트. 출력을 파일로 저장하여 컨텍스트 누적 최소화 (적용 대상: Trine Check 3.5 자동화)

- [ ] **컨텍스트 소모 모니터링 체크포인트 추가**: 루프 간격이 10분 이하이고 보고서 생성이 포함된 경우, 2시간 후 `/context` 명령으로 토큰 사용량 점검 (`trine-context-management.md` 보완 검토)

- [ ] **비개발 세션에서의 의도치 않은 루프 생성 방지**: Business 워크스페이스 환경 변수에 스케줄링 비활성화 옵션 설정 여부 검토 (적용 대상: Business `.env` 설정)

- [ ] **루프 vs 외부 cron 역할 분리 원칙 문서화**: 3일 이하 세션-내 단기 폴링 → `/loop`, 7일 이상 장기 자동화 → 외부 cron. `scripts/weekly-report/` 및 `daily-system-review`와의 역할 분리 기준 명문화

## 7. 시스템 적용 맥락

| 영상 제안 | 현재 상태 | 갭 | 우선순위 |
|----------|----------|-----|---------|
| `/loop N분마다 상태 확인` 패턴으로 CI/리뷰 폴링 자동화 | `pr-code-review-gate.md`에 `/loop 2m` 규칙 이미 명시됨 | 정책 존재하나 컨텍스트 소모 가이드 없음 | 높음 |
| 커스텀 스킬과 루프 조합 ("20분마다 /review-pr 실행") | `pr-code-review-gate`에 수동 폴링 루프 존재 | 루프+스킬 조합 패턴이 Skills 문서에 예시로 없음 | 중간 |
| `cron_list`로 활성 루프 현황 파악 | 현재 루프 관리 규칙 없음 | 복잡한 세션에서 어떤 루프가 활성화됐는지 파악 어려움 | 중간 |
| 3일 초과 필요 시 예약 작업(Scheduled Tasks)으로 전환 | `scripts/weekly-report/`의 외부 cron으로 장기 자동화 처리 중 | `/loop`와 외부 cron 역할 분리 기준이 명문화되지 않음 | 낮음 |
| 루프 컨텍스트 누적으로 인한 `/compact` 연계 | `trine-context-management.md`에 컨텍스트 70% 기준 존재 | 루프 실행 중 컨텍스트 소모 트리거 조건이 규칙에 없음 | 높음 |
| 환경 변수로 스케줄링 전체 비활성화 | 비활성화 옵션 언급됨, 현재 설정 미확인 | Business 환경에서 의도치 않은 루프 생성 리스크 관리 안 됨 | 중간 |

**시스템별 적용 메모:**
- **Claude Code Skills**: `/loop` 내장 스킬은 커스텀 스킬 목록에 추가 불필요(이미 내장). 커스텀 스킬과 루프 조합 패턴을 Skills 예시 문서에 추가 가능
- **cron (외부)**: `scripts/weekly-report/` 외부 cron과 `/loop` 내장 기능의 역할 분리 — 외부 cron은 서버 지속성 필요 작업, `/loop`는 세션-내 단기 폴링으로 구분
- **Trine Hooks**: `/loop`는 Hook과 다른 레이어 — Hook은 이벤트 기반(git commit 등), `/loop`는 시간 기반 폴링
- **NanoBanana/Stitch**: 직접 연관 없음. 단, 루프로 이미지 생성 상태를 주기적으로 확인하는 패턴에 응용 가능
