## 4. 비판적 분석

### 주장 1: "2025년 12월을 기점으로 모델 능력에 단계적 도약이 있었다"
- **근거**: Andrej Karpathy, Greg Brockman 등 저명 인사들의 발언 인용; 커서의 300만 줄 브라우저 구축 실험, Anthropic의 C 컴파일러 자율 제작 실험
- **근거 유형**: 경험적 (업계 인사 증언) + 사례적 (특정 실험 결과)
- **한계**: 구체적인 벤치마크 수치나 정량적 비교 없음. 어떤 모델 버전에서 도약이 일어났는지 명시하지 않음. 자막 오류로 인해 정확한 모델명 확인 불가.
- **반론/대안**: "도약"은 체감적 표현일 수 있음. 실제로는 점진적 개선이 누적되어 특정 임계점을 넘었을 가능성이 높으며, 모든 유형의 장기 작업에서 동일한 도약이 나타나는지는 불명확함.

### 주장 2: "범용 도구가 전문 도구보다 장기 실행 에이전트에 더 효과적이다"
- **근거**: Vercel 사례 (단일 배치 명령 전환 후 성공률 80% → 100%), Anthropic도 유사한 결론 도달
- **근거 유형**: 실증적 (Vercel 내부 실험 결과)
- **한계**: Vercel의 특정 use case(텍스트-to-SQL)에서 나온 결과를 일반화하고 있음. 도메인마다 결과가 다를 수 있음. 재현 가능한 공개 실험이 아님.
- **반론/대안**: 매우 도메인 특화된 작업(예: 의료 코딩, 법률 문서)에서는 전문 도구가 더 효과적일 수 있음. 범용 도구 접근이 유효하려면 모델 기반 추론 능력이 충분히 강해야 한다는 전제 조건이 있음.

### 주장 3: "하네스 엔지니어링이 향후 6-12개월간 가장 큰 기회 영역이다"
- **근거**: Open Clue의 폭발적 성장 사례, 특정 버티컬을 위한 always-on 에이전트 수요 논리
- **근거 유형**: 의견적 (발표자의 시장 예측)
- **한계**: 구체적인 시장 규모 데이터나 독립적 분석 없음. 6-12개월이라는 타임라인에 대한 근거가 없음.
- **반론/대안**: 규제(특히 AI 에이전트의 자율 행동에 대한 법적 책임), 보안 문제, 비용(장기 실행 에이전트의 API 비용) 등 실제 도입 장벽을 과소평가하고 있을 수 있음.

### 주장 4: "에이전트가 작업 상태를 이해할 수 있는 '읽기 쉬운 환경'이 핵심이다"
- **근거**: Anthropic 실험에서 init.sh + progress.txt 도입 후 개선, OpenAI의 문서 목차(AGENTS.md) 구조 적용 사례
- **근거 유형**: 경험적 (실험 결과)
- **한계**: "읽기 쉬운"의 구체적 정의와 측정 기준이 없음. 어떤 수준의 문서화가 충분한지에 대한 가이드라인 부재.
- **반론/대안**: 문서 품질도 결국 인간이 만들어야 한다는 점에서 bottleneck 발생 가능. 또한 시간이 지남에 따라 문서와 코드의 동기화 문제가 발생할 수 있음.

## 5. 팩트체크 대상

- **주장**: "Vercel이 전문 도구를 단일 배치 명령 도구로 교체하자 성공률이 80%에서 100%로 올랐고 속도는 3.5배, 토큰은 37% 절감되었다" | **검증 필요 이유**: 인상적인 수치이나 영상 내 출처(Vercel 블로그 제목/URL)가 명시되지 않음. 해당 블로그 포스트가 실제로 존재하는지, 수치가 정확한지 확인 필요 | **검증 방법**: Vercel 공식 블로그에서 "text-to-SQL agent" 또는 "tool simplification" 키워드로 검색하여 원문 확인

- **주장**: "Cursor가 GPT-5.1을 사용하여 300만 줄의 코드로 브라우저를 자율 구축하는 실험을 공개했다" | **검증 필요 이유**: 자막이 자동 생성이므로 "GPT 5.1"이 실제 모델명인지, 브라우저 구축 실험의 세부 내용이 정확한지 불확실. "GPT-5.1"이라는 모델은 공식 발표된 바 없음 | **검증 방법**: Cursor 공식 블로그·X(트위터) 계정에서 2025년 12월~2026년 1월 기간 관련 실험 공지 검색

- **주장**: "Open Clue(Open Claude로 추정)가 2026년 가장 큰 패러다임 전환을 열어준 최초의 프로젝트다" | **검증 필요 이유**: 자막 오류로 인해 실제 프로젝트명이 불명확함. "Open Clue"가 GitHub 오픈소스 프로젝트인지, 특정 제품인지 확인 필요. "Open Claude"나 다른 always-on 에이전트 프로젝트일 수 있음 | **검증 방법**: GitHub에서 "open-clue" 또는 "always-on agent" 키워드 검색; 채널 영상 설명란에서 원본 링크 확인

## 6. 실행 가능 항목

- [ ] **[1인 개발자/AI 빌더] init.sh 패턴 도입**: 장기 에이전트 작업 시작 시 개발 서버 설정, progress.txt 생성을 자동화하는 init.sh 스크립트 작성. Claude Code 또는 커서 기반 작업에 즉시 적용 가능.

- [ ] **[1인 개발자] AGENTS.md를 목차로 구성**: 프로젝트 내 거대한 단일 CLAUDE.md/AGENTS.md 대신, 아키텍처/DB 스키마/제품 스펙/프론트엔드 계획 등을 별도 파일로 분리하고 AGENTS.md는 목차 역할만 부여.

- [ ] **[AI 시스템 설계자] 도구 수 줄이기 실험**: 현재 에이전트에 부여된 전문 도구를 절반으로 줄이고 단일 배치 명령 도구로 통합하는 실험. 성공률·속도·토큰 소비를 비교 측정.

- [ ] **[프로덕트 빌더] 버티컬 특화 always-on 에이전트 기회 탐색**: 자신의 도메인(이커머스, SaaS, 콘텐츠 등)에서 현재 수작업으로 하는 반복 워크플로우를 목록화하고, cron 기반 자율 에이전트로 자동화할 수 있는 top 3를 선정.

- [ ] **[개발자] E2E 테스트 도구를 에이전트에 직접 제공**: 유닛 테스트만 제공하는 대신 Playwright MCP 또는 Chrome DevTools Protocol을 에이전트 환경에 포함하여 실제 브라우저 기반 검증이 가능하게 설정.

- [ ] **[시스템 설계자] 기능 목록 문서 + pass/fail 상태 패턴 적용**: 큰 프로젝트를 200개 이상의 세부 기능으로 분해하고, 각 기능에 pass/fail 상태를 부여하는 JSON 파일을 에이전트 환경에 포함. 에이전트가 항상 전체 진행 상황을 파악하고 최우선 작업을 선택하도록 강제.

| 기능 목록 문서 + pass/fail 상태 패턴 | Trine의 Spec + traceability matrix + session-state가 유사 역할 | 에이전트가 세션 시작 시 자동으로 잔여 기능 목록을 읽고 우선순위를 결정하는 루프 미구현 | **Medium** — Trine Check Gate와 결합하면 효과적 |
| E2E 테스트 도구 에이전트 제공 | Playwright CLI 도구가 설치됨 | Check 3에서 E2E 테스트는 수동 트리거. 에이전트가 자율적으로 브라우저 검증하는 시나리오 미활성화 | **Medium** — playwright-cli 스킬과 Check 3 연동 강화로 대응 가능 |
| 범용 도구 신뢰 (모델에게 최대 컨텍스트 제공) | Trine의 Haiku(탐색), Sonnet(구현), Opus(판단) 계층화로 일부 대응 | 모델에게 범용 도구(Bash, Glob, Grep, Read)를 충분히 활용하도록 하는 반면, 과도한 스킬 호출 유도 가능성 | **Low** — 현재 방향성과 일치. 지속적으로 스킬 수 최적화 필요 |
