## 1. TL;DR

Claude Code의 신규 Subagents 기능을 활용해 메인 에이전트가 E2E 테스트 시나리오 분석(1번 에이전트)과 Playwright 기반 테스트 코드 작성(N번 병렬 에이전트)을 위임하는 멀티 에이전트 워크플로우를 시연한다. 서브 에이전트는 `.claude/agents/` 폴더에 마크다운 파일로 정의하며, 컨텍스트 격리와 병렬 실행이 핵심 이점이다.

8. [🕐 15:30](https://youtu.be/u4_AcW96_78?t=930) **실제 데모 결과 — 병렬 브라우저 실행**: 3개 서브 에이전트가 동시에 여러 브라우저를 열어 각자 테스트 코드를 작성하는 모습을 시연. 순차 처리 대비 속도가 크게 향상된다고 설명한다.

9. [🕐 17:00](https://youtu.be/u4_AcW96_78?t=1020) **병렬 실행의 한계 — 의존성 충돌**: 병렬로 실행한 에이전트들이 동일 서비스(메모 앱)에서 동시에 생성/삭제 작업을 하면서 데이터 상태 충돌이 발생했다. 메모 개수를 체크하는 로직에서 서로 간섭하여 테스트가 계속 실패하는 문제가 드러났다.

10. [🕐 18:40](https://youtu.be/u4_AcW96_78?t=1120) **멀티 에이전트 운영 능력의 중요성**: "20-30개 에이전트를 동시에 멀티에이전트로 돌리면서 서비스를 개발하는 사례가 나오고 있다. LLM과 에이전트 성능의 한계가 있어 무조건 여러 개가 효율적이진 않지만, 앞으로 개발자의 역할에서 여러 에이전트를 다루는 능력은 필수가 될 것"이라는 저자의 전망.

## 4. 비판적 분석

### 주장 1: "서브 에이전트 컨텍스트 격리로 토큰 효율성과 코드 품질이 동시에 향상된다"

- **근거**: 메인 에이전트의 불필요한 컨텍스트가 서브 에이전트에 전파되지 않아 각 에이전트가 특화 작업에 집중한다고 설명.
- **근거 유형**: 경험적 주장 (실제 측정 데이터 없음)
- **한계**: 서브 에이전트를 스폰하는 오버헤드(새 컨텍스트 초기화, 도구 호출 비용) 비용은 언급하지 않는다. 격리가 항상 이득인지, 어느 임계점부터 효율적인지 수치 없음.
- **반론/대안**: 단순 작업의 경우 서브 에이전트 스폰 오버헤드가 오히려 느릴 수 있다. Trine에서 이미 Haiku/Sonnet 분리로 비용 절감을 실현 중인데, 서브 에이전트가 추가로 어느 정도의 절감을 주는지 비교 실험이 필요하다.

### 주장 2: "병렬 서브 에이전트로 E2E 테스트 작성 속도가 크게 빨라진다"

- **근거**: 3개 시나리오를 순차가 아닌 병렬로 처리하므로 이론상 1/3 시간으로 완료된다고 암시.
- **근거 유형**: 이론적 추론 (실제 시간 측정 없음)
- **한계**: 데모에서 의존성 충돌로 인해 테스트들이 지속적으로 실패하는 문제가 발생하여, 실제로 속도 이득보다 디버깅 비용이 더 컸을 가능성이 있다. 또한 영상에서 완전히 성공하는 결과를 보여주지 않는다.
- **반론/대안**: 병렬 에이전트가 공유 상태(DB, 파일)에 동시 접근할 때 격리 전략 없이는 오히려 느려질 수 있다. 각 에이전트가 독립된 테스트 데이터 픽스처를 사용하도록 설계해야 한다.

### 주장 3: "멀티 에이전트 운영 능력이 미래 개발자의 필수 역량이다"

- **근거**: 20-30개 에이전트를 동시에 운영하는 사례가 등장하고 있다는 언급.
- **근거 유형**: 의견 (출처 불명확)
- **한계**: 현재 LLM 성능 한계로 과도한 병렬화가 오히려 비효율적임을 스스로 인정하고 있다. "언제 병렬화가 적절한가"에 대한 기준이 없다.
- **반론/대안**: 단순히 에이전트 수를 늘리는 것보다, 태스크 분해와 의존성 분석(Wave 기반 스폰) 능력이 더 중요하다. Trine 워크플로우에서 이미 의존성 그래프 기반 병렬화를 실천하고 있다.

### 주장 4: "description 필드로 서브 에이전트 자동 호출이 가능하다"

- **근거**: description 필드에 기술된 내용과 유사한 작업이 들어오면 Claude Code가 자동으로 해당 서브 에이전트를 호출한다고 설명.
- **근거 유형**: 기능 설명 (Anthropic 공식 문서 기반으로 추정)
- **한계**: 자동 호출의 정확도(얼마나 잘 판단하는지)에 대한 검증 없음. 잘못된 에이전트가 호출되거나 필요한 에이전트가 호출되지 않는 경우에 대한 대응 방법 미언급.
- **반론/대안**: 중요한 워크플로우에서는 명시적 호출이 더 안정적이다. description 기반 자동 호출은 편의성이지만 프로덕션 파이프라인에서는 예측 불가능성을 높인다.

## 6. 실행 가능 항목

- [ ] **[즉시] `.claude/agents/` 디렉토리 구조 확인** (적용 대상: 모든 Claude Code 프로젝트): 현재 Trine 프로젝트에서 기존 `AGENTS.md` 방식과 신규 `.claude/agents/*.md` 방식의 차이를 공식 문서로 확인한다.

- [ ] **[단기] 의존성 없는 E2E 테스트 병렬화 패턴 적용** (적용 대상: Portfolio 프로젝트 E2E 테스트): Playwright E2E 테스트를 도메인별로 분리하여(auth / blog / search) 서브 에이전트 병렬 실행 시 데이터 충돌이 없도록 각 에이전트가 독립 픽스처를 사용하게 설계한다.

- [ ] **[단기] 서브 에이전트 마크다운 파일 형식 검토** (적용 대상: Trine 시스템): `name`, `description`, `model`, `color` 등 지원 필드 목록을 공식 문서에서 확인하고, Trine 에이전트 파일(`.claude/agents/` 파일)과 이 영상의 서브 에이전트 파일 형식이 동일한지 비교한다.

- [ ] **[단기] 병렬 스폰 시 명시적 키워드 패턴 도입** (적용 대상: Trine 병렬 실행 규칙): "병렬로"라는 키워드 없이 서브 에이전트를 호출하면 순차 실행될 수 있음. Trine Wave 기반 병렬 스폰 프롬프트에서 명시적 병렬 지시 패턴을 표준화한다.

- [ ] **[중기] Check 3 (E2E) 자동화에 서브 에이전트 적용 검토** (적용 대상: Portfolio 프로젝트 Check 3): 현재 Trine Check 3에서 수동으로 실행하는 E2E 테스트를, 시나리오 분석 서브 에이전트 + 테스트 작성 서브 에이전트 조합으로 자동화할 수 있는지 파일럿 실험.

- [ ] **[참고] Playwright MCP `.mcp.json` 프로젝트 범위 설정 확인** (적용 대상: Portfolio): "Claude Code에서 프로젝트별로 MCP를 추가하기 위해선 루트 경로의 `.mcp.json`에 추가"라고 언급. 현재 Portfolio 프로젝트의 `@playwright/cli` 설정과 `.mcp.json` 방식의 차이점 확인.

## 7. 시스템 적용 맥락

| 영상 제안 | 현재 상태 | 갭 | 우선순위 |
|-----------|-----------|-----|---------|
| 서브 에이전트로 E2E 시나리오 분석과 테스트 코드 작성을 분리 | Trine Check 3에서 E2E는 수동 실행 또는 CI 의존 | 서브 에이전트 기반 E2E 자동화 파이프라인 없음 | 중 |
| `.claude/agents/*.md`로 전문화된 에이전트 정의 | Trine agents 폴더는 `~/.claude/trine/agents/`에 별도 관리 | 프로젝트별 `.claude/agents/` 활용 패턴 미정립 | 중 |
| `description` 필드 기반 자동 호출 | Trine은 명시적 스킬 호출(`/skill-name`) 방식 사용 | 자동 라우팅 vs 명시적 호출 선택 기준 없음 | 낮음 |
| 시나리오 단위 병렬 서브 에이전트 스폰 | Trine Wave 기반 병렬 Subagent 스폰 (의존성 그래프 활용) | E2E 시나리오를 Wave 단위로 분해하는 패턴 미구현 | 중 |
| MCP를 서브 에이전트 내에서 도구로 명시적 지정 | Portfolio에서 Playwright CLI를 npm global로 사용 | 서브 에이전트별 MCP 도구 격리 설정 검토 필요 | 낮음 |
| 병렬 에이전트 데이터 충돌 방지 (테스트 픽스처 격리) | Trine 병렬 실행 규칙에 파일 소유권 선언 있음 | DB 상태 공유 시나리오에 대한 테스트 격리 가이드 없음 | 높음 |

- **Trine 병렬 실행 규칙**: `isolation: "worktree"` + 의존성 충돌 방지 패턴이 이 영상의 문제(데이터 충돌)를 이미 커버하고 있음 — 서브 에이전트 E2E 병렬화 시에도 동일 원칙 적용
